# OMEGA Core — Архитектура движка

## Структура папки `core/`

```
core/src/
├── neurons/   gX.ts          ← 1 bit = 1 нейрон
├── centers/   Li.ts          ← центры обработки данных
├── omega/     Omega.ts       ← центр истины
├── networks/  TextNet · ImageNet · VideoNet · AudioNet · GameNet
├── training/  parameters.ts  ← 35 параметров обучения
├── MirrorAI.ts               ← главный оркестратор
└── index.ts                  ← публичный API
```

---

## 1 — gX Нейрон (Mirror Neuron)

### Суть
Минимальная единица информации. **1 бит = 1 gX**.
Нейрон всегда существует как **зеркальная пара**:

```typescript
const gX1 = new MirrorNeuron(id, layer, position);  // первичное состояние
const gX2 = gX1.mirror;                              // зеркало — ВСЕГДА бит-инверсия gX1
```

| gX1 | gX2 | Смысл |
|-----|-----|-------|
|  0  |  1  | нет сигнала → есть отсутствие |
|  1  |  0  | есть сигнал → нет отсутствия |

### Почему const?
`gX1` и `gX2` — **неизменяемые константы** после создания.
Они не обучаются сами по себе — обучение происходит в Li (выбором
*каких* нейронов активировать и с какой силой).
Это как транзисторы: сам транзистор не меняется,
меняется схема их подключения.

### Зачем зеркало?
- **Встроенная коррекция ошибок**: если gX1=1 и gX2 тоже вдруг 1 — это аномалия.
- **Полное покрытие**: пара `{gX1, gX2}` описывает и наличие, и отсутствие признака одновременно.
- **Резонанс**: когда gX1 первичной сети совпадает с gX2 другой сети — это сигнал общей закономерности.

## 2 — Li Центр обработки (Processing Center)

### Суть
Li — это **обучаемый нейронный слой** поверх gX нейронов.
Он тоже зеркальный, но **изменяемый** (`let`, а не `const`):

```typescript
let Li1 = new ProcessingCenter({ type: "text", layerDepth: 6, neuronsPerLayer: 128 });
let Li2 = Li1.mirror;  // параллельная обработка с противоположным весом
```

### Как работает

```
Входные данные
      ↓
  encode()          — вектор из текста / изображения / аудио / etc.
      ↓
  Li1.process()  ──────────────────────────────────────────┐
  Li2.process()  (те же данные, обратные начальные веса)   │
      ↓                                                     │
  consensus()    ← среднее Li1+Li2, взвешенное по confidence│
      ↓                                                     │
  learn()        ← если ошибка велика → grow() добавляет слой│
      └─────────────────────────────────────────────────────┘
```

### Рост (grow)
Li **растёт автоматически** когда:
- confidence < порог несколько раз подряд
- новые данные несут знание которого ещё нет

Рост = добавление нового `NeuronLayer` со свежими gX-нейронами.

### Knowledge Store
Каждый Li держит в памяти базу знаний (`Map<key, Knowledge>`).
При каждом запросе проверяется — есть ли похожий вектор в памяти.
Если есть — ответ отдаётся **мгновенно**, без пересчёта.

## 3 — Ω Omega (Truth Center)

### Суть
Omega — финальный **синтезатор** всех Li центров.
Она тоже зеркальная (`let Omega`), но её результат становится **const**:

```typescript
let Omega = new OmegaCenter();
const truth: OmegaTruth = await Omega.synthesize(input);  // readonly, неизменяемый
```

### Алгоритм конвергенции (Mirror Loop)

```
Ω(x)          ← первичный кандидат (среднее всех Li)
Ω(Ω(x))       ← пере-оценка кандидата
Ω(Ω(Ω(x)))    ← ...

пока |Ω(Ω(x)) - Ω(x)| > threshold (обычно 2-3 итерации)
     ↓
converged = true → это и есть Истина
```

Это аналог **эхо-камеры**: ответ отражается в своём зеркале
пока не перестанет меняться — значит он стабилен.

### Зачем это нужно?
Один Li может ошибиться. Omega проверяет консенсус **всех** Li которые
участвовали в запросе и только тогда выдаёт финальный ответ.
`confidence` в результате — это мера этого консенсуса.

## 4 — Где разместить в сети для мгновенной работы

### Топология

```
Пользователь (браузер)
       │ HTTP/WebSocket
       ▼
  [Vercel Edge — Next.js]       ← apps/web  (статика, UI, роутинг)
       │ HTTP  (внутренняя сеть — НЕ через интернет)
       ▼
  [Render / RunPod — core]      ← core/src/server.ts  (Express 4000)
  ┌────────────────────────┐
  │   MirrorAI (в памяти)  │    ← загружен при старте, НЕ перезагружается
  │   ├── TextNet          │
  │   ├── ImageNet         │
  │   ├── VideoNet         │
  │   ├── AudioNet         │
  │   └── GameNet          │
  │        ↓               │
  │   Li Clusters (RAM)    │    ← Knowledge Map в process.memory
  │        ↓               │
  │   Ω Truth Center       │
  └────────────────────────┘
       │
       ▼
  [Supabase PostgreSQL]         ← только долгосрочное хранение

  [Render — Harvesters]         ← services/ (отдельный процесс, фоновый)
       │ вектора данных
       └──────────────────────▶ Li.learn() через REST
```

### Правила скорости

| Уровень | Где живёт | Почему быстро |
|---------|-----------|---------------|
| gX нейроны | **process heap (RAM)** | бит-операции, O(1) |
| Li Knowledge Map | **process heap (RAM)** | `Map.get()` = мгновенно |
| Li.process() | **CPU, параллельно** | `Promise.all([Li1, Li2])` |
| Omega loop | **CPU, 2-3 итерации** | обычно <5 мс |
| Supabase | **PostgreSQL** | только логирование, не блокирует ответ |
| Harvesters | **отдельный сервис** | никогда не блокируют запрос |

### Критические принципы

1. **Никогда не читать из базы во время запроса** — Supabase только для аналитики.
   Все знания должны быть в `Li.knowledgeBase` (RAM) к моменту запроса.

2. **Pre-warm при старте** — `MirrorAI` загружает веса из Supabase **один раз** при `npm start`.
   После этого — полностью в памяти.

3. **Harvesters асинхронны** — они кормят `Li.learn()` в фоне через очередь,
   никогда не блокируя ответ пользователю.

4. **Один процесс = один сервер** — для <1000 req/s core держится на одном Pod.
   При перегрузке → горизонтальное масштабирование через RunPod (каждый Pod со своей копией весов).

5. **WebSocket вместо REST для чата** — позволяет стримить токены по мере генерации,
   без ожидания полного ответа (perceived latency → 0).

### Рекомендуемые ресурсы

| Нагрузка | Core | RAM |
|----------|------|-----|
| Dev / тест | 1 vCPU | 512 MB |
| Продакшн (обычный) | 2 vCPU | 2 GB |
| Видео / игры (GPU) | RunPod A100 | 80 GB VRAM |
